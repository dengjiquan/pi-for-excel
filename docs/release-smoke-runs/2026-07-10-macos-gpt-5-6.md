## Transport finding and fix

The original Excel failure was transport-specific, not a model-ID or metadata failure. ChatGPT's forced SSE route rejected Luna, while native Pi's Node `auto` path succeeded. The durable WebView adaptation now:

1. Sends an authenticated browser HTTP request to the configured, allowlisted proxy.
2. Converts that request to the exact ChatGPT Codex WebSocket endpoint and wraps it as `response.create`.
3. Matches native Pi's upstream user-agent; forwarding a browser/proxy UA selected an unavailable Luna rollout alias.
4. Preserves native Pi UUIDv7 session semantics by stably mapping existing UUIDv4 app sessions for the bridge.
5. Converts upstream JSON frames back to valid SSE for the Office WebView and stops on terminal events.
6. Requires explicit proxy and WebSocket-bridge health headers and fails with deterministic upgrade instructions when unavailable.
7. Emits an SSE `error` event when an upstream stream closes before a terminal event, so partial output cannot look successful.

The bridge remains loopback/client-policy constrained, uses the existing strict target allowlist, and permits this transport only for the exact `https://chatgpt.com/backend-api/codex/responses` target. A real Chromium cross-origin fetch from `https://localhost:3141` read both capability headers, covering browser CORS exposure in addition to the server tests.
